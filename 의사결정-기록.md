# 넥스트스텝 의사결정 기록 (Decision Log)

기획부터 API v3까지, 검증한 것·결정한 것·틀려서 고친 것을 시간순으로 남긴다. 스펙 문서들은 "무엇을 만드는가"만 담고 "왜 이렇게 됐는가"는 여기 담는다. 심사 질문 방어 근거이자, 나중에 같은 조사를 반복하지 않기 위한 기록.

## 1. 아이템 선정

배민스타트업스퀘어·우아한테크코스·경기창조경제혁신센터 3자 주최 취지에 맞춰 후보를 비교한 결과 "자리 폐업 연대기"로 확정. 경쟁 후보(리뷰 답글 코파일럿, 마감 떨이 알림, 노쇼 예약금, 지원사업 매칭)는 각각 레드오션 확인·시장 미스매치 확인 후 컷. 판단 근거: 2026-01 행안부 인허가 195종 데이터 통합 개방 직후라 "왜 지금"이 성립하고, 배민(외식업 데이터)·스타트업스퀘어(예비창업자 타깃)·경기센터(창업 생존율) 삼자 결이 전부 직격.

### 3.3 물건 분리 정확도 보강 (enrichment 모듈 단계)

지번 접두사로 조회는 "묶어서"(행정동 단위, `storeListInDong`), 매칭(조인)은 "정확한 키로"(지번+상호명 일치) — 조회 단위와 조인 정밀도를 분리한 게 핵심 판단. 근거: 조회를 PNU 단위로 쪼개면 개발계정 일 1,000건 쿼터가 즉시 소진되고, 조인을 prefix로 뭉뚱그리면 다른 필지가 잘못 합쳐지는 과합침 위험이 생김.

파이썬 참조 구현(`enrichment/`)에서 실측으로 잡은 버그 2건:
- 지번 정규식이 "서울특별시"형 주소만 통과하고 "경기도 성남시"형(도+시 2단계)은 실패 — 성남 데이터 자체가 안 뚫리는 치명적 버그였음, 옵셔널 논캡처 그룹으로 수정
- fixture JSON 배열이 두 번 닫혀 파싱 자체가 실패 — 테스트가 즉시 잡아냄

## 5. 화면 설계 정정 — v2 → v3 (스크린샷 기반)

**무슨 일이 있었나**: v2에서 "주변상권 정보"를 건물 상세(화면②)의 `neighborhood` 객체로 설계했음(반경조회 집계: totalStoreCount·categoryBreakdown·recentOpenCount 등 6필드). 이후 실제 구현 화면 스크린샷을 확인한 결과, 해당 정보는 건물 상세가 아니라 **물건 상세(화면③)** 안에, "가게 자세히 보기" 드롭다운으로 선택한 개별 이력(Tenancy) 옆에 붙는 카드였음이 드러남.

**같이 드러난 두 번째 문제**: 위치만 틀린 게 아니라 필드 자체도 안 맞았음. 스크린샷 카드의 실제 필드는 전용면적·보증금/월세·권리금·일평균유동인구·주변같은업종·주변공실률 6개였는데, 이 중 우리가 확보한 소스(인허가+상가API)로 실제 만들 수 있는 건 **"주변 같은업종 개수" 1개뿐**. 스크린샷 자체에 "실 데이터 연동 전 예시값입니다" 캡션이 있었던 게 이 지점을 정확히 가리킴.

**정정 내용**:
- API 계약: `GET /api/sites/{pnu}`에서 `neighborhood` 제거, `GET /api/units/{unitId}`의 `timeline[]` 각 항목에 `marketInfo` 신설(`api-spec-v2.md` → `api-spec-v3.md`)
- 프론트: 화면② `NeighborhoodPanel` 제거, 화면③에 `TenancySelector`+`LicenseInfoCard`+`MarketInfoCard` 신설, 카드형 `TimelineItem`을 이미지와 같은 가로 바(`TimelineBar`)로 교체(`frontend-spec-v2.md` → `frontend-spec-v3.md`)
- 백엔드: 6.1절에 "정정 이력" 섹션을 명문화, `marketInfo`의 5개 목업 필드는 "실제 연동 시도하지 말 것"을 스펙에 직접 박음(`10-backend-spec.md`)
- DDL: site-level `neighborhood_snapshot` 폐기 → 이력 단위 `tenancy_market_info` + 반경조회 원자료 캐시 `site_radius_store_cache`로 재설계(`schema-v2.sql` → `schema-v3.sql`)

이 정정의 교훈: 문서 스펙만으로 화면 구조를 확정하지 말고, 실제 구현/스크린샷이 나오면 그것을 최상위 근거로 재검토해야 한다. 문서와 실물이 어긋나면 실물이 이긴다.

## 6. marketInfo 필드별 소스 최종 조사

v3에서 "실값은 sameCategoryNearbyCount 1개뿐"이라고 판정한 근거를 여기서 확정한다.

| 필드 | 조사 결과 | 판정 |
|---|---|---|
| sameCategoryNearbyCount | 상가API 반경조회(`storeListInRadius`)로 산출 가능 | **실값** |
| leaseAreaSqm(전용면적) | 국토교통부_상업업무용 부동산 **매매** 실거래가 API(data.go.kr 15126463) 존재. 단 아래 6.1 사유로 신뢰도 낮음 | 목업 유지(근거 있는 보류) |
| depositKrw / monthlyRentKrw / keyMoneyKrw(보증금·월세·권리금) | 상가 **임대차** 실거래 공개 API **자체가 없음**. 확정일자 제도(상가건물임대차보호법)는 있으나, 이 정보는 오픈API가 아니라 이해관계인이 신분증 들고 관할 세무서에 서면으로 신청하는 민원 절차(「상가건물 임대차계약서상의 확정일자 부여 및 임대차 정보제공에 관한 규칙」) | **확정 — 소스 없음, 목업 불가피** |
| dailyFloatingPopulation(유동인구) | 조사 범위에서 성남 대응 소스 미발견(서울 열린데이터광장은 지역 불일치) | 목업 유지 |
| vacancyRatePercent(공실률) | 상동 | 목업 유지 |

### 6.1 매매 API로도 전용면적을 못 채우는 이유

- **조회 파라미터가 지번주소가 아니라 "법정동코드 5자리 + 계약년월"**임. 우리가 원하는 "이 지번 하나를 넣으면 그 자리 정보가 나온다"는 조회 방식 자체가 지원되지 않음 — 동 전체 목록을 받아 텍스트로 매칭을 시도해야 함.
- 응답 안내에 "개인정보보호를 위해 상업업무용 부동산 중 일반건축물의 지번정보는 **일부만 제공**"이라 명시돼 있어, 설사 매칭을 시도해도 신뢰도가 낮음.
- 매매 실거래가 있으려면 그 물건이 **실제로 그 달에 거래**돼야 함 — 대부분의 물건은 해당 기간에 매매 이력이 없어 데이터 자체가 없음.
- 결론: 매매 API의 존재가 "전용면적을 채울 수 있다"를 의미하지 않음. 이번 스코프에선 이 필드도 목업 유지가 합리적.

## 7. 최종 스코프 판단 요약

- **핵심 기능(인허가 히스토리)**: 전량 실데이터. 프로젝트의 본체.
- **보강 필드(industryDetail/locationSource/enrichmentSource)**: 상가API로 실값, 부분 보강(폐업 이력은 원천적으로 미보강).
- **marketInfo.sameCategoryNearbyCount**: 상가API로 실값.
- **marketInfo 나머지 5필드**: 확정적으로 소스 없음, 목업+`isPlaceholder:true`+"실 데이터 연동 전 예시값" 캡션 상시 노출로 정직하게 방어.

이 구성은 심사 Q2(데이터 정확도)·Q7(부분 실패/목업 처리)의 실제 근거 자료이기도 하다 — "안 되는 걸 되는 척하지 않고, 됐다/안 됐다를 필드 단위로 구분해서 보여준다"는 것 자체가 이 서비스의 신뢰성 주장과 일치한다.

## 8. 경매정보 연동 — 소스 조사 (2026-07-18)

법원경매 이력을 자리(PNU) 단위로 곁들이는 신규 기능을 검토하며 소스를 조사함. 상세는
`server/docs/superpowers/specs/2026-07-18-auction-data-integration-design.md` 참고, 여기는 요약과
재조사 방지용 핵심 사실만.

**핵심 사실**:
- 대법원 법원경매정보(courtauction.go.kr)가 모든 상업 경매 서비스(옥션원=구굿옥션, 지지옥션, 태인,
  탱크옥션 등)의 원천 — 이들 전부 공식 API 없이 자체 크롤러로 추정(관련 특허 검색 결과로 뒷받침).
  개발자 대상 API를 파는 곳은 CODEF(유료 데이터 중계)가 유일.
- courtauction.go.kr은 **robots.txt 자체가 없음**(404 — 명시적 금지도 명시적 허용도 아닌 회색지대,
  이용약관도 검색으로는 확인 못 함, 확정 아님).
- 사이트가 **WebSquare**(전자정부 표준프레임워크 XML 렌더링) 기반이라 단순 HTML 파싱 불가 —
  헤드리스 브라우저 렌더링 또는 백엔드 AJAX 엔드포인트 역공학이 필요. 지금까지 다뤄온 정적 CSV
  다운로드(LOCALDATA)와는 난이도가 다른 부류.

**검토한 5개 접근(A~D, 전부 트레이드오프째로 설계 문서에 기록해둠 — 하나 막히면 다음 것 바로 시도)**:
CODEF 계약(A) / 헤드리스 브라우저(B1) / AJAX 역공학(B2) / 스키마만 먼저(C) / 스코프 축소+저빈도
배치(D). 처음엔 B2+D로 방향을 잡았으나, 같은 세션 안에서 `claude-in-chrome`으로 실제
courtauction.go.kr을 조작해보며 **B2가 막히고 B1이 그냥 됨을 확인해 B1+D로 뒤집음**(2026-07-19) —
AJAX 요청/응답을 페이지 컨텍스트에서 가로채려는 시도가 claude-in-chrome 자체의 쿠키/세션 유출 방지
안전장치에 걸렸고, 반대로 실제 UI를 폼-채우기→검색→텍스트 읽기로 그냥 조작하니 상세페이지까지
포함해 필요한 데이터가 전부 나옴(성남 수정구+상업용 필터 기준 실측 5건, 기일내역·감정평가요항표까지).

## 9. 담배소매업 인허가일자 신뢰성 이슈 — 미해결 기록 (2026-07-19)

**트리거**: 실사례 `4113111600002700009-U1`(PNU 금토동 270-9)에서 `씨유 성남대왕판교로점`(담배소매업)의
테넌시가 `1999-01-15 ~ 2026-01-14`(27년) 통짜 한 건으로 뜸. 사용자가 "폐업신고가 뒤늦게 붙어서
그런 거 아니냐"고 의심해 조사.

**확인된 사실**:
- 원본 `licensed_business_record`에 이미 이 기간 그대로 담긴 **단일 행**(id=102134) — 우리 파이프라인의
  병합/중복적재 버그 아님.
- 씨유(CU) 브랜드는 2012년 이후 명칭(그 전엔 훼미리마트) — `licensed_at='1999-01-15'`는 "이 상호가
  개업한 날"이 아니라 **"이 자리의 담배소매업 인허가가 최초 발급된 날"**로 보임. 담배소매업 같은
  일부 업종은 실제 운영 상호·프랜차이즈가 바뀌어도 인허가 자체(그리고 그 인허가일자)가 갱신되지
  않고 승계되는 경우가 있는 것으로 추정.
- 실측(브랜드 런칭연도보다 이른 `licensed_at` 비율): 씨유(2012~) **10.1%**(503/4,960건),
  GS25 **5.6%**(133/2,395건), 이마트24(2017~, 舊위드미) **5.2%**(68/1,302건) — 한 건짜리 특이값이
  아니라 10건 중 1건꼴로 나오는 흔한 패턴.

**검토했다가 기각한 방향 — `NoStorefrontSubCategories`에 담배소매업 추가**: 처음엔 이 클래스의
"층/호 정보 없음 90%+" 기준에 담배소매업이 들어맞는다고 판단해 추가했었으나, **부정확한 측정
때문이었음**(grep이 컬럼 경계를 안 지켜서 98.1%로 잘못 나옴). 실제로는 2026-07-18에 91,772건을
전수분석해 **84.5%**(90% 임계값 미만)로 이미 정밀 측정돼 있었고, 그래서 의도적으로 무점포 목록에서
빠져 있었다(`NoStorefrontSubCategoriesTest.담배소매업은_무점포가_아니다` 참고). 되돌림 — 카테고리
전체를 배제하면 문제없는 나머지 84.5%(정상적인 개별 편의점 레코드)까지 같이 사라진다. **애초에
"층/호 정보 없음" 기준 자체가 이 문제(licensed_at 신뢰성)를 겨냥한 게 아니었다** — 층/호 정보가
멀쩡히 있는 담배소매업 레코드도 이 날짜 문제를 똑같이 가질 수 있음(별개의 신호).

**상태: 미해결, 기록만**. 카테고리 단위가 아니라 레코드 단위(브랜드명-런칭연도 매핑 등)로 걸러야
하는데, 그런 매핑을 일반화해서 유지보수 가능하게 만드는 방법이 아직 없어 이번엔 구현하지 않음.
다음에 다룰 때 참고할 후보:
- 브랜드명(씨유/GS25/이마트24/세븐일레븐 등) → 알려진 런칭연도 매핑 테이블을 만들어 `licensed_at`이
  그보다 이르면 "표시 신뢰도 낮음" 플래그만 붙이는 방향(레코드 제외는 아님) — 다만 매핑 테이블
  자체를 누가/어떻게 유지보수할지가 관건.
- 또는 담배소매업처럼 "인허가가 상호 변경과 무관하게 승계되는" 것으로 알려진 업종에 한해,
  `licensed_at` 대신 같은 자리의 다른(비담배소매업) 레코드가 있으면 그 시작일을 참고하는 보정
  로직 — 단 이번 실사례(`4113111600002700009` 블록 없는 접미어 그룹)처럼 곁에 참고할 다른 레코드가
  없는 경우엔 보정 자체가 불가능.

## 10. DB를 H2 인메모리 → MySQL로 전환 (2026-07-20)

**트리거**: 인허가 데이터가 158만 행 규모로 커지면서, §7(backend-spec.md, 이전판) 명시대로
"기동마다 시드 SQL 재적재" 구조의 비용이 실질적 병목이 됨 — 배포마다 부팅 ~90초, jar 354MB
(시드 SQL 1,589개 파일·570MB가 jar에 통째로 포함됨). 사용자가 이 문제를 직접 지적하며 개선 요청.

**핵심 확인 사항**: `spring.sql.init.mode`만 바꿔서는 "신규 값 들어올 때만 로딩"이 안 됨 — DB가
인메모리인 이상 재시작하면 어차피 다 날아가므로 "기존 데이터가 남아있다"는 전제 자체가 성립하지
않는다. 즉 영속 DB(MySQL) 도입은 증분 로딩을 가능하게 하는 전제조건이었고, 이게 jar 용량·부팅
시간 문제도 같이 해결했다.

**결정**: 배포 서버에 MySQL 8을 직접 설치(전용 DB `turbom` + 전용 계정, 외부 미노출). 시드는
`mysql` CLI로 한 번만 적재하고, 앱 기동에서는 완전히 분리(`spring.sql.init.mode: never`). 신규
시드 청크는 `scripts/mysql/load-new-chunks.sh`가 `seed_files_applied` 추적 테이블 기준으로 아직
안 올라간 파일만 배포 파이프라인에서 자동 적재. Flyway/Liquibase 등 무거운 마이그레이션 도구는
안 씀(YAGNI — 시드는 append-only라 순서 보장이 필요 없고, 스키마 자체도 안 바뀜).

**테스트는 전혀 안 건드림**: `PersistenceSmokeTest`의 정확한 행수 검증 등 16개 테스트 클래스
전부 지금처럼 H2로 돈다. `src/test/resources/application.yml`을 새로 만들어 기존 H2 설정을
그대로 복사해두면 Maven이 테스트 클래스패스에서 이걸 메인 설정보다 우선시켜서 테스트 코드는
한 줄도 안 고쳐도 됨.

**실측 결과**: `mvn clean test` 79/79 통과. MySQL 행수(1,587,341)가 H2와 일치. 신규 jar를
MySQL 환경변수로 포그라운드 기동해 부팅 **9초**, `/api/sites/search` 실검색 정상 확인 후 배포.
jar 354MB → 239MB(단, 191MB는 Playwright 브라우저 바이너리 번들이라 이번 변경과 무관 — "jar가
수 MB로 줄어든다"는 최초 기대는 부정확했음, 실측으로 정정).

**배포 중 발견·수정한 사고**: 사전 검증 단계에서 포그라운드로 띄운 확인용 jar 프로세스가 `kill`에도
불구하고 포트 8080을 계속 붙잡고 있어서, CI가 배포한 신규 jar(systemd 관리)가 포트 바인딩 실패로
크래시 루프 중이었는데 CI의 "Verify" 스텝은 (계속 살아있던) 그 leftover 프로세스에 우연히 curl이
성공해 그린으로 통과함. `ps`/`ss -ltnp`로 실제 포트 소유 프로세스를 확인해 leftover를 정리하고
systemd가 정상적으로 포트를 넘겨받는 것까지 확인 후 완료로 판단.

**상태**: 완료, 운영 반영됨. 커밋: `turbom-server` `d1bff4b`. 상세: `backend-spec.md` §7,
롤백 절차는 배포 서버의 `/home/ubuntu/turbom-server/target/server-0.1.0.jar.bak`(H2 내장 이전
버전) 참고.

## 11. 업종(BusinessType) 도메인 재설계 — 설계 확정, 구현 대기 (2026-07-20)

**계기**: 실데이터를 다루다 세 케이스가 지금 구조(`TenancyQueryService`의 절차적 로직 +
`NoStorefrontSubCategories` 정적 Set)로는 처리하기 어렵다는 게 확인됨 — 집단급식소/위탁급식영업처럼
같은 물건에서 서로 다른 인허가로 뜨는 페어링, 매장은 있는데 인허가에 상세주소가 안 담긴 레코드,
businessStatus/licensed_at을 못 믿는 업종(담배소매업). 세부 설계·아키텍처는
`server/docs/superpowers/specs/2026-07-20-business-type-domain-design.md`.

**§9와의 관계 — 설계 초안에서 실수를 하나 잡음**: 처음 설계할 때 "담배소매업은 상세주소가 대체로
없다"는 전제로 카테고리 단위 분류를 만들려다, §9를 다시 읽고서야 이게 이미 실측·기각된 접근이라는
걸 확인함(84.5%는 90% 임계값 미만이라 `NoStorefrontSubCategories`에서 의도적으로 뺐던 값이고,
카테고리 통짜 배제는 정상 호실 정보가 있는 나머지 15.5%까지 잘못 묶음). 그래서 최종 설계는 이
축(§9가 부른 "위치미특정")을 **BusinessType의 카테고리 속성이 아니라, `LocationIdentity`가
레코드 단위로 판정**하는 걸로 바뀜 — `NoStorefrontSubCategories`의 47개 카테고리 목록과 그 의미는
전혀 안 바뀜(`BusinessType.locationCertainty()`로 값 그대로 이관).

**reliabilitySignal은 §9의 미해결 후보 중 하나를 일반화**: §9가 "다음에 다룰 때 참고할 후보"로
남겨둔 두 방향(브랜드-런칭연도 매핑 테이블 / 같은 자리 다른 레코드로 시작일 보정) 중 후자를
카테고리 한정이 아니라 일반 메커니즘(`BusinessType.reliabilitySignal(LocationContext)`)으로
구현하는 설계. §9가 스스로 지적한 한계(곁에 참고할 다른 레코드가 없으면 보정 불가)는 이번에도
그대로 남음 — 완전 해결이 아니라 부분 개선이고, 브랜드-런칭연도 매핑 테이블 쪽은 여전히 미해결.

**상태**: 브레인스토밍·설계 확정, 아직 구현 전(writing-plans 단계로 진행 예정). 코드 변경 없음.

## 12. 캐노니컬 문서 전면 드리프트 점검 — `schema.sql` 재작성 + API 필드 보강 (2026-07-20)

**계기**: 11장 작업 중 `의사결정-기록.md`/`backend-spec.md`를 제대로 안 읽어서 실수한 걸 계기로,

## 15. 주소 검색을 단순 substring에서 토큰 AND 매칭으로 전환 (2026-07-22)

**계기**: 사용자가 `GET /api/sites/search`의 실제 구현을 직접 지적 — `query` 전체를 하나의
substring으로 `jibunAddress`/`roadAddress`에 `LIKE '%query%'` 매칭만 하고 있어서 (1) 토큰
순서가 바뀌거나 중간 토큰(구/동 등) 하나만 빠져도 매칭 실패, (2) 숫자만 검색하면 지번이 아닌
층/호수 등 무관한 숫자에도 걸리는 문제가 있었다. "백엔드가 이미 PNU 단계에서 법정동코드·동명·
본번/부번을 파싱해두고 있는데(`parsedFloor` 등과 같은 부류) 그 구조화된 데이터를 안 쓰고 원문
문자열 substring만 비교하는 거라 정공법은 쿼리를 토큰 단위로 쪼개서 AND 매칭하는 것"이라는
방향을 사용자가 직접 제시.

**설계 판단**: `parsedFloor`/`parsedUnitNo` 같은 구조화된 파싱 필드에 매칭시키는 대신, 더 단순한
"쿼리 토큰화 + `jibunAddress`/`roadAddress` 원문에 대한 AND substring 매칭"으로 구현했다. 이유:
파싱 필드 매칭은 검색어가 "성남시 수정구"처럼 시/군/구·동 명칭일 때는 안 맞고(파싱 필드는
층/호 전용), 원문 주소 전체에 토큰 AND를 거는 편이 검색어 형태를 가리지 않고 더 범용적이다.
숫자 토큰만 별도로 앞뒤 숫자 경계 체크(`(?<!\d)토큰(?!\d)`)를 둬서 층/호수 오탐만 걸러냈다.

**구현**: `AddressQuery`(`domain.site`, JPA 비의존) 신설 — `anchorToken()`(DB 사전 필터용 최장
토큰, AND 매칭의 필요조건이라 후보가 안 샘)과 `matchesAll()`(애플리케이션 레이어 최종 판정).
`TenancyQueryService.searchSites()`에서 `recordRepository.searchByAddress(anchorToken)`으로 DB
후보를 좁힌 뒤 `matchesAll()`로 최종 필터링. PNU 정확 일치는 토큰 매칭과 별개로 항상 통과
(PNU는 사람이 읽는 주소 텍스트에 literal로 안 들어있어서 토큰 매칭으로 못 잡음 — 이 escape
hatch를 빼먹으면 PNU 코드 검색이 전부 깨짐, 구현 중 자체적으로 발견해 추가).
