## 프로젝트 소개

'터봄(Turbom)'은 "터를 본다"는 한국의 오래된 표현에서 시작된 이름이다. 새 가게를 열기 전 좋은 터를 고르고 개업 전 고사를 지내며 성공을 기원하던 문화에서, 창업을 준비하는 사람이 계약하려는 자리(터)의 역사를 분석해 본다는 의미로 가져왔다.

터봄은 상가 계약 전, 그 자리의 과거 개업·폐업 이력과 생존 통계를 확인하는 예비 창업자용 입지 실사 서비스다. 정부 공공데이터(인허가 데이터)를 기반으로 "어떤 업종이 운영됐는가 · 얼마나 오래 운영됐는가 · 몇 번 폐업했는가 · 동일 업종이 반복 실패했는가"를 보여준다.

**서비스 목적**: 단순 데이터 조회가 아니라 "이 자리를 계약해도 괜찮을까?"라는 질문에 객관적 판단 근거를 제공하는 것. 사용자는 서비스를 떠날 때 "계약 결정을 내릴 근거를 얻었다"고 느껴야 한다. **타겟은 예비 창업자(카페·음식점 등)**.

## 프롬프트

넥스트스텝(터봄) 프론트엔드를 만들어줘. 지번 단위 상가 자리의 개업–폐업 이력을 보여주는 예비창업자용 입지 실사 도구다. "중고차 사고이력 조회의 상가 버전".

### 역할·원칙

Stripe, Notion, Linear 수준의 SaaS를 설계하는 Senior Product Designer이자 Senior Frontend Engineer로 임한다.

- 데이터를 나열하지 말고 인사이트를 먼저 보여준다.
- 사용자는 5초 안에 핵심 내용을 이해할 수 있어야 한다.
- 숫자는 카드(Card) 형태로 시각화한다.
- 금융 서비스·정부 데이터 리포트 수준의 신뢰감을 준다. 불필요한 장식은 최소화.
- 반응형 웹(모바일·태블릿·데스크톱)으로 설계한다.
- 인터랙션은 Hover·Fade·Skeleton Loading·Card Elevation·Smooth Transition 정도로. **과도한 애니메이션은 쓰지 않는다** — 단, 랜딩 스크롤스냅·전환 커버 같은 이미 검증된 nextstep-client 인터랙션(아래 `## 검색 페이지` 참조)은 예외로 유지한다(이미 검증됐고 브랜드 경험의 일부).

### 스택 (고정)

- React 18 + Vite + TypeScript
- React Router (3개 라우트)
- TanStack Query (서버 상태)
- Tailwind CSS + **shadcn/ui** (신규 채택 — 아래 "UI 컴포넌트" 절 참조)
- react-leaflet + OpenStreetMap 타일 (지도, 키 불필요. 추후 Naver Maps SDK로 교체 예정이므로 지도 컴포넌트는 교체 가능하게 분리해둘 것)
- 배포 타깃: Vercel
- 상태 관리는 TanStack Query(서버 상태) + `useState`/`useSearchParams`(화면 상태)로 충분 — 전역 클라이언트 상태 라이브러리(Zustand 등)는 이 앱 규모에 불필요, 도입하지 않는다.

### 목/실 전환 (필수 구조)

- `VITE_API_BASE_URL` 환경변수. 미설정이면 목(mock) 모드로 완전 동작, 설정 시 실 API.
- `src/api/client.ts`에서 분기: 목 모드는 300ms 지연 후 `src/mocks/`의 고정 JSON 반환, 실 모드는 `fetch(BASE_URL + path)`.
- 함수 3개: `search(query)`, `getSite(pnu)`, `getUnit(unitId)`. 시그니처 목/실 동일.
- **목 데이터는 `frontend-spec.md` 6장의 JSON을 그대로 복사**해서 `src/mocks/`에 넣어라. 자리 2곳·물건 4개·이력 다수 포함, "시흥동 123" 검색은 의도적 빈결과.
- 데모 모드는 실제 API를 쓰는 것과 동일한 사용자 경험을 제공해야 한다 — 로딩 지연(300ms)·에러/빈결과 케이스까지 목 모드에서 그대로 재현.

### 도메인

```
Site(자리, pnu) 1:N Unit(물건, unitId) 1:N Tenancy(이력)
```

조회 식별자는 PNU. 지번 검색 → PNU → 물건 → 이력.

## 3화면·3라우트

### ① 랜딩 `/`

**nextstep-client의 `Home.tsx`가 이미 검증된 레퍼런스 구현**이다 — 뷰포트 3구간 스크롤스냅(휠 입력을 자체 이징으로 가로채는 `useEasedSnapScroll` 포함) 구조를 그대로 포팅/유지한다:

1. **Hero + 검색** — 다크 네이비 배경, 청록 포인트. Badge("창업예정자를 위한 입지 실사 리포트") · 메인카피("자리를 보면, 창업이 보입니다.") · 서브카피 · 검색바(placeholder "예: 금토동 405") + 예시 칩 3개(금토동 405 / 판교로 22 / 시흥동 123) · 3단계 안내 카드(지번검색→점포선택→히스토리확인). 검색 실행 시 같은 섹션 안에서 결과 리스트로 전환(지번주소 강조·도로명 보조·`점포 N · 폐업 M` 배지, 클릭 시 ② 화면으로 슬라이드 전환).
2. **왜 터봄인가** — 3개 Feature Card(계약 전에 확인합니다 / 자리 단위 리포트입니다 / 의사결정의 근거를 남깁니다) + 4개 Fact(공공데이터·지번단위·층호단위·운영이력).
3. **제공하는 분석 정보** (신규 — 디자인 프롬프트 반영) — 메인카피 "하나의 자리, 여섯 가지 각도." 아래 표로: 운영 이력 / 운영 기간 / 폐업 통계 / 업종 적합성 / 현재 상태 / 계약 체크리스트 각 항목 한 줄 설명. 심플한 리스트 형태.
4. **분석 리포트 미리보기 + CTA** — 실제 물건 상세와 같은 항목 구성의 샘플 카드(SAMPLE 뱃지), "계약 전에, 자리를 먼저 보세요" CTA 배너.

### 검색 페이지 (② 건물 상세, `/map?q=&pnu=`)

*(리버스 프롬프팅 — nextstep-client `Map.tsx`를 그대로 서술. 디자인 프롬프트의 "랜딩→검색페이지→지번입력→부번선택→층호목록→선택→보고서" 단계형 서술 대신, 지도 중심의 단일 화면 UX로 대체한다. 구성요소는 동일하게 충족: 검색창·지번주소 후보·상가 목록·지도·층호 선택.)*

전체화면 지도(`react-leaflet`, CARTO Positron 라이트 타일) 위에 플로팅 UI를 얹는 구조:

- **상단**: 브랜드(클릭 시 랜딩으로 커버 전환) + 검색바(포커스 시 폭 확장, `useSearchParams`의 `q`와 동기화).
- **지도**: `search(q)` 결과 중 좌표 있는 후보만 마커로 표시(인라인 SVG 핀, 선택된 마커만 확대). 여러 마커 있으면 `fitBounds`로 전부 보이게, 하나면 그 지점으로 확대. 지도 빈 곳 클릭 시 아래 패널 축소.
- **좌측(모바일: 하단) 플로팅 패널**(글래스모피즘, `bg-white/95 backdrop-blur`):
  - 후보가 2개 이상이면 **세그먼트 탭**(슬라이딩 하이라이트, 지번주소 끝 2단어를 라벨로) — 이게 "지번주소 종류 결과" 역할.
  - 선택된 자리 요약: 지번주소(강조)·도로명(보조)·`점포 N개`/`폐업 이력 M건` 배지.
  - **물건(층·호) 목록** — `getSite(pnu).units`를 폐업 많은 순 정렬. 각 항목: `label`(좌측 상태색 보더) · 상태뱃지(영업/공실) · `locationSource` 배지(인허가표기/상가API매칭/추정분리) · 현재가게명(+`industryDetail` 있으면 병기) · 미니통계(`N곳 거쳐감 · 폐업 M번 · 평균 K개월`). 클릭 → 커버 슬라이드 전환 후 `/units/:unitId`.
  - disclaimer 하단.
  - 모바일에서 패널은 지도 탭으로 끄트머리만 남기고 접힘, "점포 목록" 플로팅 버튼으로 재전개.
- 로딩(스켈레톤)/빈결과(`'{query}' 주소의 기록을 찾지 못했어요`)/에러(재시도) 상태 전부 패널 안에서 처리.
- 화면 전환은 순간 컷이 아니라 색 커버가 화면을 덮은 뒤 라우팅(랜딩行은 페이드, 상세行은 아래에서 슬라이드업).

### ③ 물건 상세(보고서) `/units/:unitId` (핵심 화면)

`getUnit(unitId)` 호출. 정보는 **디자인 프롬프트가 정한 순서를 유지**한다 — Summary·위험도·Insight가 Timeline/세부데이터보다 먼저 나와야, 사용자가 5초 안에 판단할 수 있다:

1. **상가 정보(Header)**: `unit.label` + 주소, 현재 업종(`industryDetail ?? subCategory`), 현재 운영기간, 관측기간(`yearsSpan()`), 우측에 종합 위험도 카드(별점+등급 텍스트).
2. **핵심 요약(Summary)**: 위험도 / 최근 폐업 횟수 / 평균 생존기간 / 현재 업종 / 현재 운영기간 / 동일 업종 수(`sameCategoryNearbyCount`, 실값) / 반경 내 점포 수(신규 — 아래 "리포트 확장 필드" 참조). Metric Card 형태.
3. **종합 분석**: 사람이 쓴 리포트처럼 자연스러운 문장 3~4줄. **"AI"라는 표현은 절대 쓰지 않는다.**
4. **주변 상권 분석**(신규 — 아래 "리포트 확장 필드" 참조): 업종 구성(Horizontal Bar Chart) · 경쟁도(0~100 게이지) · 상권 통계(동일업종수/최근개업수/전체점포수/집계기준일) · 상권 특징(Tag).
5. **위험도**: 별점 1~5 + 등급(매우안정~매우위험) + 게이지 + 종합 해석.
6. **핵심 인사이트**: 4개 Signal Card(아이콘+수치+제목+짧은설명).
7. **운영 이력**: 하나의 넓은 Timeline 카드 안에 좌(과거)→우(현재) 가로 바. 구간 클릭 시 상단에 "가게명·업종·N개월" 배지. 업종 색상 범례 하단(대분류가 여럿이면 대분류로, 전부 같으면 소분류로 구분 — nextstep-client `Unit.tsx`의 `colorKeyOf` 로직 그대로).
8. **통계**: 좌(자리 운영: 폐업횟수/평균생존/최장/최단/동일업종실패) · 우(주변 상권: 전체점포/동일업종/최근개업/반경/집계일) 2-column 카드 그리드. `동일 업종 실패`·`집계일`처럼 강조 필요한 항목은 wide card.
9. **가게 자세히 보기**: 드롭다운(`TenancySelector`, `tenancyId` 선택, 기본값=최근 이력) + 2단 카드 — 좌(인허가정보: 상호명/category/subCategory/개업일/폐업일/영업기간/영업상태), 우(`MarketInfoCard`: 헤더 "예시" 뱃지 고정, 전용면적/보증금·월세/권리금(0="무")/일평균유동인구/주변같은업종 N곳/주변공실률, 하단 "실 데이터 연동 전 예시값입니다" 캡션 상시).
10. **계약 체크리스트**: 진행률(Progress) + 8개 체크항목 — 최근폐업횟수/평균생존기간/동일업종실패여부/현재운영기간/경쟁점포수/최근개업흐름/업종분포/상권밀도.
11. **CTA**: "좋은 창업은 여러 자리를 비교하는 것에서 시작됩니다" + "새로운 자리 분석하기" 버튼(랜딩으로).

disclaimer 하단 상시. 상태: 로딩 / 404 / 이력0.

## 검증된 로직 재사용 (insights.ts 포팅)

3·5·6·10번 섹션은 새로 설계하지 말고 nextstep-client `src/lib/insights.ts`를 그대로 포팅한다(통계·타임라인에서 파생 계산하는 1차 휴리스틱, 검증됨):

- `riskLevel(closedCount)` → 별점 1~5 + 등급 텍스트. 위험도 카드·헤더에 사용.
- `repeatCategoryFailure(timeline)` → 동일 category 2회 이상 폐업 여부. 종합분석·체크리스트 문장에 사용.
- `signals(statistics, timeline, todayISO)` → Signal 4개(최근 5년 폐업 흐름 / 현재업종 운영기간 / 평균 생존기간 / 평균 대비 운영기간). 핵심 인사이트 섹션 그대로 사용.
- `diagnosis(statistics, timeline, todayISO)` → 종합분석 문장 배열.
- `checklist(statistics, timeline, sameCategoryNearbyCount, todayISO)` → 현재 5개 항목. **디자인 프롬프트는 8개를 요구**하므로 아래 "리포트 확장 필드"의 목업 값을 받는 파라미터를 추가해 경쟁점포수/최근개업흐름/업종분포/상권밀도 3~4개 항목을 보강한다.

기존 로직은 `category`만 참조하던 부분을 이번 스펙 정합화로 `category`/`subCategory` 2단 구조에 맞게 그대로 이식(타입만 맞추면 로직 자체는 안 바뀜).

## 리포트 확장 필드 (신규 — api-spec.md에 없는 목업)

디자인 프롬프트의 "주변 상권 분석"·"통계(주변 상권)"·"핵심요약(반경 내 점포수)"·"체크리스트(경쟁점포수 등)"가 요구하는 다음 필드들은 **현재 확정된 `api-spec.md`의 `marketInfo`(실값은 `sameCategoryNearbyCount` 하나뿐)에 없다**. 데이터셋·상가API로 실측 불가능하다는 게 이미 `의사결정-기록.md` §6에서 확정된 사실이라, 지금 백엔드를 확장하지 않는다 — **차후 기능 보강 대상**이다.

지금은 다음 방침으로 목업을 만든다:

- `api-spec.md`의 `MarketInfo` 타입은 건드리지 않는다(계약 오염 방지).
- `src/types.ts`에 프론트 전용 확장 타입을 별도로 둔다(예: `MarketAnalysisMock`) — `industryBreakdown`(업종별 점포수), `competitionScore`(0~100), `totalNearbyStoreCount`, `recentOpenCount`, `districtTags`(string[]). "집계기준일"은 새 필드를 만들지 말고 기존 `marketInfo.asOf`를 그대로 재사용(같은 시점 기준이라는 걸 보여주는 게 목적이라 굳이 중복 필드 불필요).
- 이 목업의 **실제 숫자 콘텐츠는 손으로 대충 채우지 말고, 지금 가진 mock 데이터셋(timeline·category 분포 등)을 근거로 그럴듯하게 도출**해서 채운다 — 이미 있는 `sameCategoryNearbyCount`·업종 분포와 숫자가 어긋나지 않도록(예: "동일업종 14곳"인데 "전체점포 3곳"처럼 모순되면 안 됨).
- 화면엔 반드시 `MarketInfoCard`와 동일한 원칙으로 "예시" 뱃지/캡션을 노출해 실데이터처럼 보이지 않게 한다 — `marketInfo.isPlaceholder` 캡션과 같은 톤("실 데이터 연동 전 예시값입니다").
- 나중에 백엔드가 실제로 이 필드들을 지원하게 되면, 그때 `api-spec.md`에 정식으로 추가하고 이 mock 전용 타입은 제거한다.

## 타입 (그대로 사용)

`프론트-연동계약서.md`의 TypeScript 인터페이스 블록을 `src/types.ts`로 그대로 가져가라. 특히 **업종 3필드 구분** 주의:
- `category`: 인허가 대분류(예: 동물, 음식) — 항상 존재, 타임라인 기본 표시
- `subCategory`: 인허가 소분류(예: 동물미용업, 일반음식점) — 항상 존재, 물건 상세에서 표시
- `industryDetail`: 상가API 세부업종(영업중 이력만) — **있으면 subCategory 대신 우선 표시**, null이면 subCategory 사용
- industryDetail이 null인 폐업 이력은 subCategory로 표시하는 게 정상(빈 태그 만들지 마라).

## 주의사항 (데이터 현실)

- 대부분의 물건은 이력이 여러 개(연대기). 폐업 이력엔 `industryDetail`·`marketInfo` 실값이 없는 게 **정상**(상가API는 현재 스냅샷). 버그 아님.
- `closedAtEstimated`는 현재 데이터셋에서 항상 false(폐업일자 실값). 처리는 해두되 화면 영향 미미.
- `marketInfo`의 6필드 중 `sameCategoryNearbyCount`만 실값, 나머지는 목업. "리포트 확장 필드"도 전부 목업. 절대 실데이터인 것처럼 표시하지 마라 — "예시" 뱃지·캡션 필수.
- disclaimer(전체 서비스 고지)와 marketInfo/확장필드 캡션(카드 목업 고지)은 **다른 것**. 둘 다 노출.

## UI 컴포넌트 (shadcn/ui)

`npx shadcn@latest init -t vite`로 초기화(경로 alias `@/*` — `tsconfig.json`에 `baseUrl`/`paths`, `vite.config.ts`에 `resolve.alias` 추가 필요. `@types/node` devDependency 추가). 이후 `npx shadcn@latest add <component>`로 아래를 받는다:

`card` `badge` `tabs` `accordion` `alert` `dialog` `tooltip` `skeleton` `separator` `progress`

기존 nextstep-client의 수제 컴포넌트(`StatusBadge`, `Disclaimer`, `Skeleton`, `Card` 등)는 대응하는 shadcn/ui 컴포넌트로 교체. 지도·타임라인처럼 shadcn/ui에 없는 도메인 전용 컴포넌트는 계속 직접 구현. 모든 숫자는 카드 기반으로 시각화.
