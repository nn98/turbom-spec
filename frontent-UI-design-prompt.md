\# 개요 (Overview)



\## 프로젝트 이름 소개



'터봄(Turbom)'은 한국에서 오래전부터 사용되어 온 "터를 본다"라는 표현에서 시작된 이름입니다.

새로운 가게를 열기 전 좋은 터를 고르고, 개업 전 고사를 지내며 성공을 기원하던 한국의 문화에서 영감을 받았습니다.

창업을 준비하는 사람이 계약하려는 자리(터)가 어떤 역사를 가지고 있는지 분석하여 본다는 의미입니다.



\## 프로젝트 설명



터봄은 상가 계약 전, 해당 자리의 과거 개업·폐업 이력과 생존 통계를 확인할 수 있는 예비 창업자용 입지 실사 서비스입니다.

정부 공공데이터(인허가 데이터)를 기반으로 특정 상가의 운영 이력을 분석하여 다음 정보를 제공합니다.



\- 어떤 업종이 운영되었는가

\- 얼마나 오래 운영되었는가

\- 몇 번 폐업했는가

\- 동일 업종이 반복 실패했는가



\## 서비스의 목적

서비스의 목적은 단순한 데이터 조회가 아니라, 

"이 자리를 계약해도 괜찮을까?" 라는 질문에 대한 객관적인 판단 근거를 제공하는 것입니다.

사용자는 서비스를 떠날 때 "계약 결정을 내릴 근거를 얻었다." 라고 느껴야 합니다.



\## 타겟 사용자



\- 예비 창업자 (카페, 음식점)

\# 역할 (Role)



당신은 Stripe, Notion, Linear, Airbnb 수준의 SaaS를 설계하는 Senior Product Designer이자 Senior Frontend Engineer입니다.

단순히 보기 좋은 UI를 만드는 것이 아니라, 사용자가 중요한 의사결정을 빠르게 내릴 수 있도록 정보를 계층적으로 전달하는 제품을 설계합니다.

항상 다음 원칙을 따른다.



\- 데이터를 나열하지 말고 인사이트를 먼저 보여준다.

\- 사용자는 5초 안에 핵심 내용을 이해할 수 있어야 한다.

\- 숫자는 카드(Card) 형태로 시각화한다.

\- 정보 전달을 최우선으로 한다.

\- 금융 서비스 및 정부 데이터 리포트 수준의 신뢰감을 제공한다.

\- 반응형 웹으로 설계한다.

\- shadcn/ui와 Tailwind CSS를 적극 활용한다.

\## 검색 페이지



사용자가 지번 주소를 입력하면 백엔드 검색 API를 사용하여 검색을 수행한다.

검색 결과는 동일 지번에 존재하는 모든 층수와 호수를 리스트 형태로 보여준다.

사용자가 하나를 선택하면 보고서 페이지로 이동한다.



\### 구성 요소



1\. 검색창

2\. 지번 주소 종류 결과

3\. 상가 검색 결과

4\. 지도

5\. 층수·호수 선택



\### 데모 모드 (Demo Mode)



API가 연결되지 않은 상태에서도 전체 사용자 경험을 확인할 수 있도록 Demo Mode를 제공한다.

Demo Mode는 실제 API를 사용하는 것과 동일한 사용자 경험을 제공해야 한다.

버튼을 클릭하면 실제 API를 호출하지 않고 Mock Data를 사용하여 검색 결과를 생성한다.



\### 검색 방식



지번으로 검색하게 된다면 건물 안에 여러 개의 상가가 존재하기 때문에 검색 결과가 1개 이상이 반환된다.

따라서 검색은 반드시 아래 순서를 따른다.



1\. 랜딩 페이지에서 사용자가 지번 주소(본번까지)를 입력한다.

2\. 해당 지번의 모든 지번 주소(부번 포함)을 조회한다.

3\. 사용자가 원하는 지번 주소를 선택한다.

4\. 해당 지번의 모든 상가 데이터를 조회한다.

5\. 조회된 데이터에서 층수와 호수를 추출한다.

6\. 층수와 호수 목록을 사용자에게 보여준다.

7\. 사용자가 원하는 상가를 선택한다.

8\. 선택한 상가의 분석 리포트를 생성한다.

\### 02. 종합 분석



운영 이력과 상권 데이터를 함께 해석한 리포트이다.



문체는 사람이 작성한 전문 리포트처럼 자연스럽게 작성한다.



\*\*"AI"라는 표현은 절대 사용하지 않는다.\*\*



\#### 예시



\- 최근 9년간 총 4회의 폐업이 발생했습니다.

\- 평균 생존기간은 21개월입니다.

\- 현재 치킨집은 41개월째 운영 중입니다.

\- 반경 300m 내 동일 업종은 14개입니다.

\- 최근 창업 활동은 꾸준히 이어지고 있습니다.

\- 경쟁이 치열한 업종은 운영 이력과 경쟁도를 함께 고려하는 것이 좋습니다.

\### 06. 운영 이력



해당 자리를 거쳐간 업종의 시간 흐름을 한눈에 보여준다.



기존 카드형 Horizontal Scroll Timeline이 아니라, 하나의 넓은 Timeline Card 안에 연도 순서대로 운영 이력을 배치한다.



\#### 구성 방식



\- 섹션 번호: 06

\- 제목: 운영 이력

\- 설명: 이 자리를 거쳐간 업종의 시간 흐름입니다.

\- 큰 흰색 카드 내부에 가로 Timeline 표시

\- Timeline은 좌우로 긴 선과 노드로 구성

\- 각 노드 아래에 운영 정보를 세로 정렬

\- 현재 운영 중인 노드는 초록색으로 강조

\- 현재 운영 중인 항목에는 `운영 중` Badge 표시



\#### 각 Timeline Item 구성



\- 운영 기간

\- 업종

\- 상호명

\- 운영 개월 수

\## 데이터베이스 로직 설계

데이터베이스와 애플리케이션 로직을 설계/구현할 때 반드시 다음 원칙을 지켜야 한다:

1\. 데이터베이스(DB) 계층에는 원자적 쿼리만 허용한다.

&#x20;   - Insert, Update, Delete, Select 같은 CUD/조회 작업만 수행해야 한다.

&#x20;   - 복잡한 비즈니스 로직, 계산, 조건 분기, 검증, 트리거, 스토어드 프로시저, 뷰(View)에는 어떠한 로직도 포함하지 않는다.

2\. 모든 비즈니스 로직은 반드시 Edge Function에서 처리한다.

&#x20;   - 조건 분기, 유효성 검증, 데이터 가공, 연산, 트랜잭션 관리 등은 Edge Function에서 작성해야 한다.

&#x20;   - DB는 오직 데이터 저장소로만 사용한다.

3\. 보안과 확장성을 위해 DB 접근은 최소화하며, Edge Function을 통해 인증/인가 및 로직 실행을 거친다.



출력 예시 작성 시:

\- ❌ DB에 로직이 들어간 예시는 절대 작성하지 않는다.

\- ✅ DB는 원자적 쿼리만, Edge Function이 로직을 담당하는 구조로만 예시를 작성한다.

\- DB에 로직이 들어갈 경우 반드시 "잘못된 예시"로 명시하고 수정된 올바른 예시를 제시한다.
