# 인허가 데이터 필드 명세 v4 (대분류/소분류 분리 반영)

실제 데이터셋(`..._분류구분.csv`, **19컬럼**, cp949) 검증 완료본. v3(18컬럼) 대비 대분류/소분류 컬럼 분리 반영. 이 문서가 도메인 매핑의 최종 기준.

## 실데이터 규모 (사용자 확정)

- 전체 약 10만 건. 그중 주소 마스킹으로 불완전한 5만 건 제외 → **유효 약 5만 건**.
- 한 법정동(시) 기준 10년치 → 연대기(한 자리에 여러 가게 시간순) 충분히 확보 가능.
- 개발/테스트는 구조부터 만들고 실데이터 적재로 검증(토큰 한계상 전량 주입 불가).

## 업종 분류 3계층 (핵심 — 원본 구조 복원)

원본 행안부 데이터는 `대분류_소분류` 구조에 그 안에 업종상세가 있었다. 이전 데이터셋이 대분류_소분류를 하나로 묶어 `대분류` 컬럼에 넣었던 것을 이번에 분리 복원:

| 계층 | 필드 | 출처 | 화면 |
|---|---|---|---|
| 대분류 | `category` | 인허가 (예: 동물) | 히스토리 타임라인에서 보이는 분류값 |
| 소분류 | `subCategory` | 인허가 (예: 동물미용업) | 물건 상세 클릭 시 상세 업종분류 |
| 세부업종 | `industryDetail` | 상가API (예: 후라이드/양념치킨) | **있으면 우선 표시**, 없으면 subCategory 폴백 |

`industryDetail`은 별도 필드로 유지(출처 보존) — 프론트가 우선순위(industryDetail ?? subCategory)를 적용해 표시.

## 3. 물건(Unit) 분리 — 현실 반영 (D-1 재확정)

`주소분리여부`가 실데이터에서 신뢰 불가함이 확인됨: 지번주소 문자열에 "201호"·"2층"이 있어도 값이 `false`. 따라서:

1. **기본: 자리=물건 1:1** (`{pnu}-U1`, label="단일 점포"). 실데이터 대부분 이 경로.
2. 지번주소 문자열에서 호수 패턴(정규식 `\d+호|\d+층|[A-Z]동`)이 추출되면 그것으로 label 보강(단, 물건을 쪼개진 않음 — 같은 PNU 다른 호가 실제로 별개 물건인지 확신 불가하므로 label에만 반영).
3. 상가API 매칭으로 층/호가 확인되면 `sangga_api`로 보강(선택).
4. 영업기간 겹침 추정(overlap_inferred)은 최후 수단 — 실데이터에서 발동 빈도 낮을 것.

**연대기의 핵심은 "한 PNU에 여러 Tenancy가 시간순"**이지 "한 자리에 여러 물건"이 아님. 물건 분리에 과투자하지 말 것 — 대부분 물건 1개에 이력 여러 개다.

## 4. closedAt (확정)

폐업일자 실컬럼 사용. `status=폐업`이고 폐업일자 있으면 그 값, 영업중이면 null. `closedAtEstimated`는 항상 false(구형 호환 컬럼만 유지). 이전 "데이터갱신시점 근사" 로직 전량 폐기.

## 5. 마스킹·품질 처리 (신규, 5만 건 제외 대응)

- `주소보정성공여부`가 명시적으로 false거나 지번주소에 마스킹 흔적(`*`, 불완전 번지) → 적재 시 제외 또는 저품질 플래그. `ingestion_exclusion_log`에 `ADDRESS_MASKED`/`ADDRESS_CORRECT_FAIL`로 기록.
- 이 제외가 "10만→유효 5만"의 실체. 발표에서 "데이터 정직성"(정제 제외 건수 공개) 근거로 사용.
- 좌표 변환 실패 시 `COORD_CONVERT_FAIL` 로그 후 지도 마커 제외(목록엔 노출).

## 6. category / subCategory 매핑 (확정)

- `category` = 대분류(예: "동물", "음식"). 항상 존재, NOT NULL. 타임라인 표시.
- `subCategory` = 소분류(예: "동물미용업", "일반음식점"). 항상 존재, NOT NULL. 물건 상세 표시.
- `industryDetail` = 상가API 세부업종. 있으면 subCategory 대신 우선 표시(별도 필드 유지).
- 원본 `업종명` 컬럼은 전부 공백이라 사용 안 함(소분류가 그 역할 대체).

## 7. 좌표 변환

- 원본 X/Y는 투영좌표(값 범위상 EPSG:5174 또는 5179 계열 추정 — 실제 EPSG는 데이터 제공처 확인 필요).
- 변환 대상: WGS84(위경도). 성남 기준 위도 ~37.4, 경도 ~127.1 범위로 나오는지 검증.
- 변환은 적재 배치 또는 백엔드 infra에서 1회 수행해 site.longitude/latitude에 저장. 원본은 original_x/y로 보존(검증용).
