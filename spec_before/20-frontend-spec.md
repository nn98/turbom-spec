# 넥스트스텝 프론트엔드 명세 (Claude Code용)

이 문서 하나로 프론트를 독립 구현한다. 상위 계약은 `00-api-contract-v1.md`. 백엔드 없이 목(mock)으로 3화면을 완성한 뒤 실 API로 스위치한다.

## 0. 목표

지번을 검색해 그 자리(건물)의 물건들을 지도·목록으로 보고, 물건을 선택해 그 자리에서 거쳐간 가게들의 개폐업 히스토리와 생존 통계를 확인하는 3화면 웹앱. 해커톤 MVP.

## 1. 스택 · 배포

- React 18 + Vite + TypeScript
- 라우팅: React Router
- 상태/서버데이터: TanStack Query(React Query)
- HTTP: fetch 래퍼(또는 axios)
- 지도: react-leaflet(확정). 타일은 OSM 기본, VWorld 타일로 교체 가능(stretch, URL만 교체). 좌표는 백엔드가 VWorld 지오코딩으로 채워 내려줌 → 프론트는 좌표를 받아 찍기만 함.
- 스타일: Tailwind CSS(확정)
- 배포: Vercel

## 2. 목 우선 개발 원칙

- `VITE_API_BASE_URL` 환경변수. 미설정이면 목 모드.
- `src/mocks/`에 계약 200 예시(3종)를 그대로 JSON으로 둔다.
- API 클라이언트는 목 모드일 때 지연(300ms) 후 목 반환, 실 모드일 때 fetch.
- 이 구조로 백엔드 완성 전 전 화면·전 상태(로딩/빈결과/에러)를 완성한다.

## 3. 라우트 · 화면

| 경로 | 화면 | 데이터 |
|---|---|---|
| `/` | ① 랜딩 · 지번 조회 | GET /api/sites/search |
| `/sites/:pnu` | ② 지도 · 조회 결과 | GET /api/sites/{pnu} |
| `/units/:unitId` | ③ 물건 상세 · 히스토리 | GET /api/units/{unitId} |

플로우: ① 검색 → 후보 클릭 → `/sites/:pnu` → 물건 카드 클릭 → `/units/:unitId`.

## 화면 ① 랜딩 · 지번 조회 (`/`)

목적: 한 줄 가치 제안 + 검색 진입.

구성
- 히어로: 서비스명 "넥스트스텝", 태그라인 "이 자리, 몇 번이나 바뀌었을까". 배경은 기획 톤(다크 네이비 + 청록 포인트).
- 검색바: 지번/도로명 입력. 플레이스홀더 "예: 금토동 405". 엔터 또는 버튼으로 검색.
- 검색 결과 후보 리스트(검색 시 등장):
  - 각 항목: 지번주소(강조), 도로명주소(보조), `물건 N개 · 누적 폐업 M건` 뱃지.
  - 클릭 → `/sites/:pnu`.
- 예시 주소 칩 3개(판교 대표 자리) → 클릭 시 바로 검색(데모 안전장치).

상태
- 초기: 히어로 + 검색바 + 예시 칩.
- 로딩: 검색바 하단 스켈레톤.
- 빈 결과: "해당 주소의 기록을 찾지 못했어요" + 입력 팁.
- 에러: 재시도 버튼.

호출: `search(query)` → `candidates[]`.

---

## 화면 ② 지도 · 조회 결과 (`/sites/:pnu`)

목적: 자리를 지도에 띄우고, 그 건물의 물건들을 골라 상세로 보낸다.

레이아웃: 좌(또는 상) 지도 + 우(또는 하) 물건 목록 패널. 모바일은 지도 위/목록 아래 세로 스택.

지도
- react-leaflet. center = site 좌표, 마커 1개(자리 위치).
- 좌표 null이면 지도 자리에 "위치 정보 없음" 플레이스홀더, 목록은 정상 노출.

상단 요약 바
- 지번주소 + 도로명주소.
- `이 건물에서 확인된 물건 N개` 요약.

물건 목록(핵심 상호작용)
- 물건 카드 반복:
  - `label`(예: 1층 101호 / 물건 A / 단일 점포).
  - 현재 상태 뱃지: 영업(청록) / 공실(회색).
  - 현재 가게명(영업 시) 또는 "현재 공실".
  - 미니 통계: `총 N개 가게 거쳐감 · 폐업 M · 평균 K개월`.
  - 클릭 → `/units/:unitId`.
- 정렬: 폐업 많은 순(임팩트 우선) 기본, 토글로 층/호 순.

disclaimer: 패널 하단에 상시 노출(계약 필수).

상태: 로딩(카드 스켈레톤) / 404(자리 없음 → 검색으로 복귀 CTA) / 물건 0개(드문 케이스, 안내).

호출: `getSite(pnu)` → `{ site, units[], disclaimer }`.

---

## 화면 ③ 물건 상세 · 히스토리 (`/units/:unitId`)

목적: 이 물건에서 거쳐간 가게들의 연대기 + 생존 통계. 서비스의 하이라이트.

상단
- 물건 라벨 + 지번/도로명 주소.
- 통계 카드 3~4개: `거쳐간 가게 N` / `폐업 M회` / `평균 생존 K개월` / `최단 L개월`. (해당 없으면 "-")

히스토리 타임라인(핵심 시각화)
- 세로 타임라인. 과거(위) → 현재(아래), 계약의 timeline은 오름차순이므로 그대로.
- 각 항목 카드:
  - 가게명 + 업종.
  - 기간: `2016.03 ~ 2018.11` (영업중이면 `2023.01 ~ 현재`).
  - 생존 개월 뱃지: `32개월`.
  - 상태 뱃지: 폐업(회색) / 영업(청록) / 휴업(주황).
  - `closedAtEstimated=true`면 "폐업일 추정" 작은 뱃지.
- 생존 개월을 막대 길이로도 표현(가로 바)해 한눈에 비교.

disclaimer: 하단 상시 노출.

상태: 로딩 / 404(물건 없음) / 이력 0건(안내 + 자리로 복귀).

호출: `getUnit(unitId)` → `{ unit, statistics, timeline[], disclaimer }`.

## 4. 공통 컴포넌트

- `SearchBar`, `AddressText`(지번 강조/도로명 보조), `StatusBadge`(영업/폐업/휴업/공실/추정), `StatCard`, `Disclaimer`, `EmptyState`, `ErrorState`, `LoadingSkeleton`, `TimelineItem`, `SurvivalBar`.

## 5. API 클라이언트

- `src/api/client.ts`: `search`, `getSite`, `getUnit` 세 함수. 목/실 스위치 내장.
- 타입: 계약 응답을 그대로 TS 인터페이스로. `Candidate`, `SiteDetail`, `Unit`, `UnitDetail`, `Tenancy`, `Statistics`, `Disclaimer`, `ApiError`.
- React Query 키: `['search', query]`, `['site', pnu]`, `['unit', unitId]`.

## 6. 반응형 · 접근성

- 모바일 우선. ② 화면 지도/목록 세로 스택.
- 색만으로 상태 구분 금지(뱃지에 텍스트 병기).
- 심사장 시연 대비: 데스크톱 큰 화면에서 타임라인이 시원하게 보이도록 최대 폭 컨테이너.

## 7. 디자인 톤

- 기획서 톤 계승: 다크 네이비(#0d1b2a) 헤더, 청록(#2f8f7d) 포인트, 강조 오렌지(#b3401e).
- 타임라인·통계가 주인공. 장식 최소, 데이터 가독성 최우선.

## 8. 완료 기준(DoD)

- 3화면 목 모드에서 전 플로우(검색→건물→물건→히스토리) 동작.
- 로딩/빈결과/에러/404 상태 각 화면 구현.
- disclaimer 3개 조회 화면 모두 노출.
- `VITE_API_BASE_URL` 설정 시 실 API로 무수정 스위치.
- Vercel 배포, 실 백엔드(Railway) 연동 확인.

## 9. 착수 순서 (Claude Code)

1. Vite+TS+Router+React Query 스캐폰딩, 목 3종 배치.
2. API 클라이언트(목/실 스위치) + 타입.
3. 화면① 랜딩·검색·후보.
4. 화면② 지도·물건 목록(react-leaflet).
5. 화면③ 히스토리 타임라인·통계.
6. 상태(로딩/빈/에러/404)·반응형·톤 마감.
7. Vercel 배포 → env로 실 API 스위치.

- **D-STYLE** (확정): Tailwind CSS.
