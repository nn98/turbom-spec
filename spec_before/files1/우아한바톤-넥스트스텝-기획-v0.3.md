## 1. 한 줄 정의

특정 상가 자리(지번 단위)의 개업–폐업 이력과 생존 통계를 보여주는 예비창업자용 입지 실사 도구.

가설: 예비창업자는 계약 전 "이 자리의 과거"를 알 방법이 없다. 인허가 공공데이터를 지번 키로 시계열 결합하면 이 정보가 만들어진다. V-1에서 소상공인365가 행정동 최소 단위까지만 조회됨을 확인 → 자리 한 곳의 연대기는 시장에 부재, 차별화 실재.

## 2. 검증 결과 (2026-07-06 실측)

### V-4 데이터 적합성 — 통과

서울시 일반음식점 인허가 샘플(41건, localdata 동일 스키마)로 4개 필드 실측:

- 폐업일자(dcbymd): 존재. 폐업건 전수 채움 확인. **단 공백 문자열(스페이스 패딩) 저장 → strip 후 빈값 처리 필수.**
- 지번주소(sitewhladdr): 100% 채움. "서울특별시 강동구 암사동 501-24 예지빌딩" 형태. 정규식으로 시/구/동/번지 분리 실측 성공.
- 폐업 이력 잔존: YES. 영업상태명=폐업 레코드가 파일에 남음 → 연대기 재료 존재 입증(프로젝트 사활 조건 통과).
- 좌표(x,y): 이 샘플은 0% 채움. 전체 파일에선 채워질 수 있으나 미확인 → 지오코딩 폴백 설계로 리스크 흡수.

결론: D-4 산식(폐업일자 − 인허가일자) 성립. 원래 설계 유지.

### 기타

- V-1 소상공인365 = 행정동 단위 조회. 자리 단위 이력 미제공 → 차별화 성립(심사 Q1 방어).
- V-2 상가(상권)정보 API 키 승인. 4~6종, 각 1만 건. 역할 = 주변 밀도 보조(자리 이력 원천 아님).
- V-3 행사장 = 성남 수정구 금토동 405-1, 판교 제2테크노밸리.
- V-5 규정: 기존 프로젝트·코드·오픈소스·외부API·AI도구 허용, 단 핵심기능은 행사 기간 중 개발. → 인프라·적재·정제는 사전, 조회·통계·화면은 당일.

## 3. 타깃 · 가치 · 수익

- 1차: 외식업 예비창업자 — 계약 전 실사, 권리금·임대료 협상 근거.
- 2차: 일반 사용자 — "그 자리 왜 자꾸 바뀌지" 호기심 유입(바이럴).
- 수익: 자리 실사 리포트 건당 과금 + 상가 중개·임대 플랫폼 제휴.
- 확장: 인허가 195종 업종으로 수평 확장(미용실·학원·약국 자리).

## 4. 데이터 소스 (역할 재편 — A/B 명확 분리)

| 소스 | 역할 | 반영 위치 | 상태 |
|---|---|---|---|
| 일반음식점 인허가(localdata 전체파일) | **자리·물건·이력 원천** | timeline, statistics | 확정 |
| 휴게음식점 인허가(localdata) | 카페·분식 커버 | 상동 | 확정 |
| 상가(상권)정보 API(소진공) — 동 단위 매칭 | **히스토리 보강** (층/호·세부업종) | units[].industryDetail 등, timeline[].industryDetail 등 | 확정 |
| 상가(상권)정보 API(소진공) — 반경조회 | **물건상세 marketInfo(주변 같은업종 개수만 실값)** | timeline[].marketInfo | 확정 |
| localdata 변동분 API | 시연 직전 최신화 | — | 후순위(V-6) |

핵심 전환(유지): 자리 이력의 원천은 상가API가 아니라 **인허가 raw의 지번주소**. 상가API는 두 갈래로만 쓴다 — 이력에 태그를 얹는 보강, 그리고 완전히 별도인 주변상권 집계. 이 둘을 같은 필드로 섞지 않는다.

## 5. 스코프

### In (당일 데모)

1. 자리 검색(지번/도로명 → PNU 후보)
2. **건물 상세**: 물건(Unit) 목록 — D-1 규칙으로 분리, 층/호·현재상태·미니통계
3. 연대기 타임라인(업소명·업종·영업기간·생존개월·현재상태)
4. 생존 통계(폐업 횟수·평균 생존개월·최장·최단)
5. **주변상권 패널**(반경 300m 전체/동종 점포수·업종분포·최근개업 추정) — 상가API 실패 시 패널만 null 처리, 나머지 화면 정상
6. 데이터 기준일·한계 상시 고지

### Out

회원·로그인·결제·PDF·전국커버리지·크롤링·리뷰.

### Stretch

경기도 인접 시 확장, 자리 2곳 비교 뷰. (LLM 총평은 D-6에 따라 제외)

## 6. 정제 규칙 (실측 반영)

- P-1 영업상태명 → {영업, 폐업, 휴업, 기타}. 기타 제외 후 집계.
- P-2 **폐업일자 공백 패딩 strip 후 빈값 판정.** 폐업인데 폐업일자 빈값 → 데이터갱신일자(updatedt) 대체 후 플래그.
- P-3 폐업일자 < 인허가일자 → 제외 후 집계.
- P-4 지번주소 정규식 파싱(시/구/동/번지) → PNU 유도. 실패 시 좌표 폴백, 좌표도 없으면 제외.
- P-5 지번주소 꼬리 건물명 제거 후 매칭. 동일 지번 재인허가 → 별개 이력 유지(업종 변경 추적이 가치).

제외·대체 건수는 발표에서 "데이터 정직성"으로 공개.

## 7. 도메인 모델 (Site—Unit—Tenancy 3계층, v0.2 대비 Unit 신설)

- Site: pnu, roadAddress, jibunAddress, point(nullable)
- **Unit**: unitId, sitePnu, label(상세주소/물건A·B/단일점포), locationSource(license\|sangga_api\|overlap_inferred)
- Tenancy: unitId, businessName, category, **industryDetail(상가API 보강, 폐업이력은 항상 null)**, licensedAt, closedAt(null=영업중), status, survivalMonths(파생), **enrichmentSource(sangga_api\|license_only)**
- UnitStatistics(파생): closureCount, averageSurvivalMonths, longestMonths, shortestMonths
- MarketInfo(tenancyId 단위, sameCategoryNearbyCount만 실값·나머지 5필드 목업+isPlaceholder) — **행이 없으면 sameCategoryNearbyCount만 null, 나머지는 상수 유지**

집계는 적재 시 배치, 조회는 읽기 전용 + 캐시. sameCategoryNearbyCount는 상가API 반경조회 결과를 자리 단위로 캐시한 것이라 요청마다 외부 호출 안 함.

## 8. API (확정 — 상세는 `api-spec.md`)

- GET /api/sites/search?query= → PNU 후보 목록
- GET /api/sites/{pnu} → 물건 목록(보강 필드 포함)
- GET /api/units/{unitId} → 연대기 + 통계(보강 필드 포함) + **이력별 marketInfo 내장**

인증 없음. 3개로 확정. **정정(스크린샷 기반)**: 당초 주변상권 정보를 건물 상세(②) 응답에 `neighborhood`로 얹는 설계였으나, 실제 화면은 물건 상세(③)의 "가게 자세히 보기" 카드였음이 확인돼 `timeline[].marketInfo`로 이전. `marketInfo` 6필드 중 실값은 "주변 같은업종 개수" 1개뿐이고 나머지 5개(전용면적·보증금/월세·권리금·유동인구·공실률)는 확보한 소스가 없어 목업(`isPlaceholder:true`)으로 표시. 근거는 `의사결정-기록.md` 참조.

## 9. GWT 시나리오

- S-1 검색: Given 적재 데이터, When 주소 검색, Then PNU 후보(주소·현재 업소 수)
- S-2 연대기: Given 자리 선택, When 조회, Then 시간순 이력(업종·기간·생존개월·상태)
- S-3 통계: Given 이력 1건+, When 조회, Then 폐업횟수·평균·최장·최단
- S-4 주변: Given 자리 선택, When 조회, Then 반경 동종 수·3년 추이
- S-5 고지: When 어떤 화면, Then 기준일·"인허가 신고 기준, 실제 영업과 차이 가능" 상시
- S-6 빈 자리: Given 이력 0건, When 조회, Then 기록없음 + 인접 자리 제안
- S-7 이상치: Given P-3 위반, When 적재, Then 제외 및 로그
- S-8 폐업일자 공백: Given dcbymd 패딩만 존재, When 적재, Then 빈값 판정·대체·플래그(P-2 회귀 방지)

## 10. 4인 분업

- A 파이프라인: 다운로드→정제(P-1~5)→적재 배치, 이상치 집계 리포트 (사전 착수 가능)
- B 도메인·조회: Site·TenancyRecord·통계 배치·API 3종 (핵심기능=당일)

- D 프론트·발표: 검색·타임라인·지도 화면, 발표자료, 데모 시나리오

글렌: 인프라 경험상 C 또는 도메인 설계상 B. 팀 결정.

## 13. 심사 예상 질문 · 방어

- Q1 소상공인365와 차이 → 상권 평균(행정동) 대 자리 한 곳 연대기. V-1 근거.
- Q2 데이터 정확도 → 인허가 신고 기준 한계 상시 고지 + 이상치 제외 건수 공개.
- Q3 수익성 → 리포트 건당 과금 + 중개 제휴. 유입은 B2C 호기심, 매출은 B2B 실사.
- Q4 왜 지금 → 2026-01 인허가 195종 통합 개방 직후, 데이터 장벽이 방금 낮아짐.
- Q5 확장성 → 195종 수평 확장, 프롭테크 접점.
- Q6 개인정보 → 공개 인허가만 사용, 대표자명 등 개인 필드 미수집.
- Q7 상권정보(상가API) 실패하면? → 부분 실패 설계: marketInfo.sameCategoryNearbyCount만 null, 나머지 화면(물건목록·지도·히스토리·인허가정보)은 정상. 목업 5필드는 애초에 외부 API 의존이 없어 실패할 수도 없음. 외부 API 의존이 서비스 전체를 막지 않는다.
